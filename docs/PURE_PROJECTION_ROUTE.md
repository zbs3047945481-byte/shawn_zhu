# 纯蒸馏投影路线说明

## 1. 为什么改成这条路线

当前项目的主目标是：

- 做成文件夹级可接入插件
- 在中小数据、强异构联邦分类场景中提升 FedAvg
- 保持轻量、低侵入、易工程化

基于这个目标，当前实现不再强调“共享特征/残差特征分离”，而是收敛到更稳妥的定义：

- 主模型中间特征 `h` 负责原始分类任务
- 插件把 `h` 投影到低维空间得到 `z_s`
- `z_s` 只承担 prototype 聚合与蒸馏正则的角色

这条路线更贴合当前实验事实，也更适合工程交付。

## 2. 当前方法如何表述

建议在论文、答辩、文档里统一使用下面的表述：

- 低维投影 prototype 蒸馏模块
- 面向强异构联邦分类任务的即插即用正则插件
- 通过类原型蒸馏抑制客户端间表征漂移

不建议继续使用下面这些表述：

- 已经完成共享特征与残差特征的严格分离
- `z_r` 已被证明为任务无关特征
- 当前方法属于完整 disentanglement 模型

## 3. 方法解释

训练时存在两条通路：

1. 主任务通路
   - `pred, h = model(X, return_feature=True)`
   - `h -> classifier -> cross entropy`

2. 插件正则通路
   - `z_s = projection_module(h)`
   - 本地按类别形成 prototype
   - 与服务端全局 prototype 做 MSE 蒸馏

总损失：

`L = L_cls + lambda * L_distill`

这里 `L_distill` 的作用不是替代分类损失，而是为主模型增加一个跨客户端的一致性约束。

## 4. 这条路线下应重点做的实验

### 主实验

- `FedAvg` vs `fedfed_prototype`
- `dirichlet_alpha = 0.1 / 0.3 / 0.5 / 1.0`
- 多随机种子重复

重点回答：

- 在强异构场景下是否稳定优于 FedAvg
- 收益是否主要出现在 `alpha` 较小场景

### 参数实验

- `fedfed_lambda_distill`
- `fedfed_sensitive_dim`
- `fedfed_noise_sigma`

重点回答：

- 额外正则强度多大最合适
- 低维投影维度是否足够轻量
- 隐私扰动和性能之间如何权衡

### 工程实验

- 插件开启前后训练链路是否保持稳定
- prototype 上传量相对模型参数量有多小
- 切换 `--plugin_name fedfed_prototype` 是否即可接入

## 5. 答辩时建议怎么说

### 如果老师问：为什么不再强调特征分离

建议回答：

> 早期版本尝试把表示拆成 `z_s` 和 `z_r`，但从当前代码与实验看，真正稳定起作用的是低维 prototype 蒸馏通道，而不是残差分量本身。为了保证方法解释和工程实现一致，我把最终版本收敛为“低维投影 + 类原型蒸馏插件”，避免做超出实验支持范围的表述。

### 如果老师问：你的方法核心贡献是什么

建议回答：

> 核心贡献不是重新设计整个联邦学习框架，而是在不改变 FedAvg 主流程的前提下，增加一个可插拔的 prototype 蒸馏正则模块。它在强异构、小数据场景中帮助客户端表示保持更一致，同时保持较低通信负担和较小接入成本。

### 如果老师问：这是不是 FedFed 思路

建议回答：

> 是 FedFed 风格思路的工程化收敛版本。它保留了“通过中间表示做跨客户端蒸馏”的核心思想，但把最终实现收敛到更容易部署的类原型蒸馏插件，而不是追求更重的表征分解模型。

## 6. 后续如果还要继续做强

如果后面想提升论文方法深度，可以考虑两条扩展线：

1. 每类多个 prototype / 聚类中心
2. 给低维投影空间增加额外结构约束

但这些都应建立在当前纯投影路线已经稳定有效的前提下。
